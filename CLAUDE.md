# txt

A fully client-side-encrypted text vault.

- **Backend (Turso, libSQL/SQLite-compatible cloud)** holds all structured data — accounts, wrapped keys, R2 object paths, sharing grants, bookmarks, and read-position — but only ever sees ciphertext, or values that are already public (public keys, KEM ciphertexts/salts).
- **Document content** lives in R2 object storage, not Turso; Turso only holds the (wrapped) R2 path.
- **All encryption, decryption, and key (un)wrapping happens client-side** (or in admin tooling), never in the database.

The schema and key hierarchy are documented in full in [docs/data_model.md](docs/data_model.md) — read it before touching anything related to the schema, key hierarchy, or sharing. Per-user isolation is real cryptographic envelope encryption — `umk` wraps everything an owner holds — with one intentional exception: `txt_shares`, which grants another user access to a specific document via asymmetric (KEM) wrapping rather than by revealing the owner's `umk`.

## CLI (`txt.py`)

Six commands beyond `--init`: `--txt-ingest <dir>`, `--txt-download <dir>`, `--txt-delete`, `--txt-delete-id <txt_id>`, `--purge-bucket`, `--txt-clean-bucket` (plus a one-time `--update-schema` for pre-existing DBs). All implementation detail — upload-before-DB-write ordering and rollback, R2 retry/cleanup, the `txt_metadata` R2-object-reuse scheme, OPF sidecar parsing — is in [docs/cli.md](docs/cli.md); read it before touching `txt/ingest.py`, `txt/download.py`, `txt/delete.py`, `txt/owner.py`, or `txt/bucket.py`.

## Web UI (`ui/`)

A second, browser-based client implementing [docs/ui.md](docs/ui.md)'s design (React + TypeScript + Vite + Vitest, Bootstrap CSS/Icons): Unlock, Library, and Reader screens over the same vault, reading (and lightly writing: read-position, bookmarks) it directly from the browser rather than through `txt.py`. See the root [README.md](README.md#web-ui) for how to run it.

- `ui/src/crypto/` ports `txt/constants.py` + `txt/crypto.py` 1:1 (`leancryptoLoader.ts` binds the prebuilt `ui/leancrypto/leancrypto.js`/`.wasm`; `blob.ts`/`kem.ts` mirror `Blob`/`Kem` exactly, cross-checked against fixtures generated by the real `txt/crypto.py`).
- `ui/src/data/` ports `txt/creds.py`/`txt/owner.py`/`txt/download.py`/`txt/r2.py` (read-only R2 access via `aws4fetch`, Turso via `@libsql/client`'s browser-safe `/web` build).
- The one thing this config doesn't carry that `admin_creds.json`/`user_cred_template.json` do is `r2_config` — the UI fetches and decrypts that from Turso's `r2_config` table with the account's `umk` instead (`ui/src/data/owner.ts`'s `fetchR2Config`), same wrapping pattern as `key_store.priv_key`.
- A `VaultContext` (`ui/src/state/VaultContext.tsx`) holds the unlocked session in memory only, never persisted, so a reload always lands back on Unlock.
- `ui/src/log.ts`'s `verbose()` (on by default, `?verbose=0` to disable — see [docs/development.md](docs/development.md)) logs `unlock()`'s steps and, via `ui/src/data/db.ts`'s wrapped `Client`, every screen's `db.execute()` calls.
- `ui/src/localIndex/` (see [docs/local_index.md](docs/local_index.md)) is the verifier bundled into `creds/local_index.html` — a separate, never-deployed file opened directly (e.g. `file://`) instead of the CDN URL, so its verifier can't be tampered away the way one shipped inside the CDN-served bundle itself could be. One accommodation this forces elsewhere in `ui/src/`: `appRouter.ts` swaps `BrowserRouter` for `MemoryRouter` under an opaque/null origin, since the History API throws there (accepted tradeoff: no address-bar updates, no back/forward or swipe-back between screens) — see docs/local_index.md's Router section for why that's detected empirically rather than by inspecting `location.protocol`/`location.origin`.

- [docs/cli.md](docs/cli.md) — `txt.py`'s commands and how they're implemented: upload/DB-write ordering and rollback, R2 retry/cleanup, the `txt_metadata` R2-object-reuse scheme, OPF sidecar parsing, bucket housekeeping.

- [docs/development.md](docs/development.md) — running Python/UI tests, and the UI's verbose logging.
